# 过滤条件

部署业务感知后，报文经过 UDG 时，会根据Rule的过滤条件进行流匹配。命中过滤条件后，对应Rule上绑定的策略生效，对报文进行处理，实现业务感知功能。

**图1** 过滤条件架构

<br>

![](过滤条件_92407888.assets/zh-cn_image_0292407903_3.png)

如 [图1](#ZH-CN_TOPIC_0292407888__fig542475914285) 所示，过滤条件允许根据用户报文的3/4层协议、协议或者协议组、7层信息进行过滤。Filter可以直接绑定在Flow Filter上，Flow Filter可以直接绑定在Rule上。但是当Filter/Flow Filter内部的逻辑较为复杂时，可以将多个Filter/Flow Filter绑定成Filter Group/Flow Filter Group。Flow Filter Group内的Flow Filter之间，可以选择与/或/非的关系。

- 与：当报文命中组内所有流过滤器时，才视为命中了该流过滤器逻辑分组。
- 或：当报文命中组内任意流过滤器时，即视为命中了该流过滤器逻辑分组。
- 非：当报文没有命中组内任意流过滤器时，才视为命中了该流过滤器逻辑分组。

过滤条件包含的参数有：

- **Filter**：3/4层过滤器，可以配置三、四层的协议、源端口号、目的端口号、源IP地址和目的IP地址等过滤条件。
- **Filter Group**：可以将过滤器配置到过滤器组，实现复杂过滤器组逻辑。
- **L7 Filter**：7层过滤器，可以配置子七层过滤器，包括URL、User-Agent、method等七层关键字段在内的各种过滤条件。可以进一步配置子七层过滤器，实现对关键字段的区分配置。通过配置7层过滤器便可以实现的应用如：[GWFD-110201 HTTP2.0 Host识别](../../../../特性指南/UDG特性指南/业务感知功能/GWFD-110201 HTTP2.0 Host识别_70518421.md)、[GWFD-110203 HTTPS业务地址识别](../../../../特性指南/UDG特性指南/业务感知功能/GWFD-110203 HTTPS业务地址识别_71023371.md)。
- **Protocol**：7层过滤器使用的应用层协议，配置内容取决于L7 Filter需要的协议，如HTTP、FTP等。
- **Protocol Group**：多个应用层协议组合。在UDG上，可以通过SA特征库预制各类应用的所使用到的协议组合，这样只需要定期更新特征库，便可轻松配置和维护多数应用使用到的各类协议组，并判断访问内容，感知业务，对不同业务进行不同的计费和动作处理。特征库支持的应用协议组合见[表1](#ZH-CN_TOPIC_0292407888__table96951541198)：

- **Flow Filter**：Filter、L7Filter、Protocol、Protocol Group的绑定组合，通过其中绑定的过滤条件，可以对用户报文实现3/4、7层的业务感知。只有过滤的内容才需要配置具体参数，不需要过滤的内容配置为Any或者不配置即可。举例而言：如果仅需要对三四层过滤，不需要对七层过滤，则不用添加L7filter。
- **Flow Filter Group**：可以将流过滤器配置到过滤器组，实现复杂流过滤器组逻辑。

另外，一些业务感知的应用，不仅需要配置过滤条件，还需要和规则属性共同完成配置，比如 [GWFD-110202 HTTP2.0协议回落](../../../../特性指南/UDG特性指南/业务感知功能/GWFD-110202 HTTP2.0协议回落_68597362.md) 。

如果需要使用IPv6协议的报文开启业务感知，则需要配置IPv6 Filter，并绑定在Flow Filter下。IPv6 Filter可以和其他过滤条件共存。详见 [GWFD-020352 IPv6 SA](../../../../特性指南/UDG特性指南/IPv6功能/GWFD-020352 IPv6 SA_80136510.md) 。
