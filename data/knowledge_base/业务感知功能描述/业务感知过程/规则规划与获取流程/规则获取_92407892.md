# 规则获取

- [规则类型](#ZH-CN_TOPIC_0292407892__1.3.2.1)
- [实现方式](#ZH-CN_TOPIC_0292407892__1.3.3.1)

规则获取流程是指PCF下发规则给SMF，并进一步通过SMF下发到UPF上的过程。

#### [规则类型](#ZH-CN_TOPIC_0292407892)

从PCF的视角看，可以下发的规则有两种：

- **预定义规则**
  预定义规则即在部署流程中规划的业务感知规则。在整个网络中，可能规划成百上千条Rule，如果不对规则的启用情况进行控制，用户一旦接入，所有规则都生效。结果是，一方面对于不同类型的用户，无法实现差异化业务感知，影响后续的计费等规划；另一方面，在用户正常会话的情况下，很多规则并不会被使用到，而业务感知本身对于硬件资源占用较大，如果对每个用户都应用所有的规则，则对设备冲击很大。因此所有的预定义规则，在部署完成后，默认都处于未启用的状态。
- **动态规则**
  动态规则与预定义规则是互补的关系。在用户激活会话、会话过程中，已部署的预定义规则无法变更，只能通过PCF控制启用/停用预定义规则。为了弥补这一缺陷，引入了动态规则。动态规则可以在用户激活或会话过程中，经过PCF下发，灵活部署到SMF、UPF上生效。
- **本地规则**
  本地规则在UPF、SMF上都会配置，当SMF从PCF处无法获取下发的规则，或网络中本身无PCF时，会启用本地规则。

在发起用户会话时，规则和策略、条件、QoS参数等很多信息一起下发。本章节聚焦于规则的获取，完整的会话建立、规则下发流程请详见 [GWFD-020351 PCC基本功能](../../../../../特性指南/UDG特性指南/智能策略控制功能/GWFD-020351 PCC基本功能_42369272.md) 和相关信令分析。

#### [实现方式](#ZH-CN_TOPIC_0292407892)

以5G核心网为例，规则下发和QoS策略下发是完全耦合、同步进行的，主要涉及4个流程：PDU会话建立、PDU会话修改、PCFP会话建立、PCFP会话修改。

**会话建立时规则下发**

如 [图1](#ZH-CN_TOPIC_0292407892__fig28481723203819) 所示，在会话建立流程中，在确认会话使用的AMF、SMF、PCF之后，PCF会下发规则经SMF最终到UPF上。

1. SMF从UDM获取签约QoS信息，并在之后向PCF发送Npcf_SMPolicyControl_Create Request消息，其中包含自身已配置的PCC规则。
2. PCF侧根据用户信息配置策略生成的“条件”，并结合SMF提供的预定义规则，生成完整的QoS策略，并将该策略通过Npcf_SMPolicyControl_Create Response下发到SMF。消息中的PccRule属性携带了规则名称，至少有动态规则、预定义规则的其中一种。
3. SMF将PCF的规则，通过PFCP Session Establishment Request消息的Create PDR信元下发到UPF。如果是动态规则，可能会根据需求同时下发Create URR、Create QER等信元。
4. UPF向SMF发送PFCP Session Establishment Response响应。
5. 之后便可以开始建立PDU会话，用户使用移动数据网络。会话过程中，UPF会根据策略中下发的Trigger，定期上报使用量、流量监测、用户状态等情况。

**会话过程中规则下发**

会话过程中，用户使用的策略可能会变更，这时候PCF会通过Npcf_SMPolicyControl_Update Response下发新的策略，并决策旧策略是否需要修改、删除。

1. 策略变更的发起方可能是UE（本身业务变更需要更新QoS）、SMF（UDM识别到本地配置策略有修改）、PCF（识别到PCF上配置的策略有修改）、基站（识别到现有QoS不能满足业务要求），但是最终都会通过信令交互，将策略变更需求传递到SMF。
2. SMF发送Npcf_SMPolicyControl_Update Request消息给PCF，向PCF上报订阅事件及相关信息。PCF根据SMF上报的会话数据和本地的用户签约数据重新进行策略判断，生成新的规则。
3. PCF通过Npcf_SMPolicyControl_Update Response消息将新的规则下发给SMF，完成SMF与PCF之间策略更新。如果没有部署动态PCC，SMF可以通过本地策略来决定是否改变QoS。
4. SMF向UPF发送PFCP Session Modification Request消息，消息中携带新的策略新信息，比如Remove PDR、Create PDR、Update PDR等信元。
5. UPF通过PFCP会话标识识别需要修改的PFCP会话上下文。之后，UPF根据SMF发送的参数列表更新该PFCP会话上下文的参数。UPF返回PFCP Session Modification Response消息，该消息包含UPF需要向SMF提供的所有信息，以响应接收到的控制信息。
