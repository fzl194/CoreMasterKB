v2.0 三层架构重构

Knowledge Mining Pipeline 架构

离线知识挖掘引擎。将原始文档经 9 阶段 Pipeline 转化为结构化知识资产（asset_core.sqlite），供 Serving 在线检索。三层单向依赖架构 + Stage Registry 热插拔版本选择。

06

距工业级的差距（诚实评估）

基于 Codex 工业级审查（2026-04-29），当前系统距真正工业级知识 ingestion 平台的关键差距。

P0 — 运行正确性（阻断级）

问题说明业界对标
并发结果与文档绑定不可靠StreamingPipeline.process_all() 不保序，可能把 A 文档结果写进 B 文档 snapshotLangGraph / Airflow: stable identity binding
run status 真相源分裂API 返回 failed，数据库却写 completed。调度、告警、恢复全部建立在错误事实之上Kubernetes: etcd 单一真相源

P1 — 发布质量门

问题说明
validate_build 只检查结构不验证 retrieval unit 质量、LLM provenance、导航污染
quality eval 未接入 release gate质量评估存在但不是发布必要条件
stage event 缺 run_document_id无法把 completed/failed 事件稳定归属到文档

P1 — 架构合同

问题说明
stage registry 未接入主链run.py 直接 import 具体实现，registry 形同虚设
Domain Pack 不是唯一合同源部分模块仍绕过 DomainProfile 私读 YAML
核心判断：当前系统在"结构分层"上前进了，但底盘还没收口——并发正确性、状态真相源、质量发布门、架构合同闭环四条线同时没有收口。继续增加能力只会放大错误产物和排障成本。
07

演进路线图

不建议继续"想到什么补什么"。按 A → B → C 三阶段收口。

Phase A: 主链可信生产线

保证同一份输入永远产出可追溯、可重跑、可判定状态的结果。

  • 修复并发结果-文档绑定
  • 统一 run status 真相源
  • 补全 stage completed event 的 run_document_id
  • 新增并发乱序写回回归测试

Phase B: 质量门成为 Release Gate

保证系统不会把劣质知识资产发布成 active release。

  • quality eval 接入 build/release 主链
  • 定义 hard gate vs warning 清单
  • 增加 golden regression 集
  • LLM provenance、navigation pollution 纳入 gate

Phase C: 平台化合同闭环

不改 core 代码，只替换 Domain Pack 就能切换场景。

  • 扩充 DomainProfile，停止模块私读 YAML
  • stage registry 真正接入主执行链
  • 默认行为从特化改为 generic baseline
  • 新增 Domain Pack 合同测试
底盘收口前，不建议继续做：增加 retrieval unit 类型、扩更多 LLM stage、堆复杂 rerank/graph/embedding 策略、强化 registry 概念层包装。原因：底盘未收口，加能力只会放大错误产物和排障成本。
