v1.2 TARGET

CoreMasterKB 技术架构

面向 Agent 的知识后端 — 不是文档搜索工具，而是一套可被 Agent 作为 Skill 调用的知识加工、精准检索与自主演化基础设施。 从原始文档摄入，到结构化知识提取，再到意图驱动的混合检索，最终输出可被 Agent 直接消费的 ContextPack。

SECTION 01

为什么需要知识后端

传统文档搜索的局限

  • 关键词盲区：用户问"UPF 怎么处理计费策略"，但文档写的是"用户面功能执行 Charging Rule"，纯关键词匹配直接丢失
  • 无结构理解：表格里的参数对照、流程步骤的先后依赖、段落之间的因果/对比关系，全部丢失
  • 静态不演化：知识只增不减，过时内容和新内容混在一起，无置信度评估，无冲突检测

Agent 时代对知识服务的三个要求

  • 精准意图理解：不是返回"相关文档"，而是理解 Agent 要"概念解释"还是"操作步骤"还是"参数对比"，返回精准 Evidence
  • 结构化证据包：不是一堆零散片段，而是带溯源链、带关系图、带置信度的完整 ContextPack
  • 持续自治演化：知识库能够自动积累高质量内容、淘汰过时内容、检测矛盾内容，越用越准
CoreMasterKB 的定位：知识加工（Mining）+ 精准检索（Serving）+ 自主演化（Evolution）三位一体的 Agent Knowledge Backend。不是给人类用的搜索引擎，而是给 Agent 用的 Skill 级知识服务。
SECTION 02

设计哲学

不是从零造轮子，而是将工业级数据工程实践融入知识系统设计，每个决策解决一个真实痛点。

设计决策工程范式解决的问题
内容快照不可变内容寻址存储相同内容通过 SHA256 哈希去重，一份快照可被多文档引用。文档身份与内容物理分离，避免重复存储
读写路径分离CQRSMining 独占写路径，Serving 纯读路径，通过 Build/Release 边界解耦。写操作永远不会阻塞读服务
阶段事件溯源Event Sourcing每个 Pipeline 阶段产出带时间戳的事件记录，任意文档的处理历史、耗时、失败原因可精确回溯
五层本体组织Hierarchical Ontology概念→机制→方法→条件→场景五层结构，每层对应独立认知框架，不同角色获取不同粒度的知识
双层图分离Dual-Layer Graph推理层（聚合语义边）与溯源层（Fact→Document 证据链）分离，图遍历和证据追溯互不干扰
声明式查询引擎Algebraic Primitivesseed / expand / combine / aggregate / project 五个代数原语，规划器自动检测依赖并行执行
五维置信度评分Multi-Factor Confidence来源权威度×提取方法×本体契合度×跨源一致性×时效性，每条知识都有可信度量化评估
知识自治演化Auto-Promote三层候选词治理 + 自动晋升门控。高质量知识自动积累到本体，低质量被淘汰，实现"自然选择"

SECTION 04 — 核心叙事

一条数据的完整旅程

追踪真实文档 业务感知场景举例.md 穿过 Mining Pipeline 全部 9 个阶段，每一步展示真实的内容变换。

追踪对象：一份电信 5G 业务感知技术场景文档。描述 SMF/PCF/UPF 协作执行业务感知策略的完整流程，包含 1 张用户场景表、4 步编号流程、10+ 技术实体（SMF、PCF、UPF、CHF、FlowFilter、PCCGroup、Rule、URR、APN、UserProfile）。预计产出 ~8 个 segments。
STAGE 1 · INGEST

文件扫描与采集

输入data/knowledge_base/业务感知功能描述/业务感知场景举例_92407896.md
检测MIME: text/markdown | SHA256(raw): a3f2b8c1... | 大小: ~1.2KB
输出RawFileData { path, mime, raw_hash, size } — 递归扫描知识库目录产出
STAGE 2 · PARSE

Markdown → 结构树

输入RawFileData（原始 Markdown 文本）
变换MarkdownParser 通过 markdown-it 将原始文本转为 SectionNode 树，保留 heading / table / list / paragraph 节点类型
输出SectionNode 树结构:
Root: "业务感知场景举例" (H1) ├─ paragraph: "以下通过一个场景举例说明业务感知的端到端生效过程..." ├─ table: 用户类型|业务类型|业务需求|策略|套餐内容 (1行数据) └─ ordered_list: 4 个步骤 ├─ Step 1: 业务感知配置过程 (FlowFilter/PCCGroup/Rule/UserProfile) ├─ Step 2: 规则部署过程 (SMF→PCF→UPF) ├─ Step 3: 用户会话中的规则匹配 (UPF 执行策略) └─ Step 4: 策略执行 (计费/重定向)
STAGE 3 · SEGMENT

结构分段 → Raw Segments

输入SectionNode 树
变换遍历树节点，每个有语义内容的节点生成一个 RawSegment，按 block_type 和 semantic_role 分类，计算 token 估算
输出
#block_typesemantic_role内容摘要~tokens

2tablereference_data用户场景表: 普通用户 | 视频业务 | 1元/Gb55

STAGE 4 · ENRICH

规则增强 → 实体提取 + 元数据

输入6 个 RawSegments

② DefaultRoleClassifier 语义角色分类 → procedure / reference_data / introduction

输出 提取的实体集合：

STAGE 5 · BUILD_RELATIONS

构建段间关系图

输入Enriched Segments（含实体标注）
变换 结构关系（v1.1 已实现）：
· previous/next: Seg#3→Seg#4 (配置→部署的先后顺序)
· same_section: Seg#3~#6 (同属一个编号列表)

语义关系（v1.2 规划）：
· elaboration: Seg#2(场景表) →elaborates→ Seg#3(配置详情)
· cause_effect: Seg#6(流量耗尽) →causes→ 策略切换(Rule3 替换 Rule2)
· temporal: Seg#3 →precedes→ Seg#4 →precedes→ Seg#5
输出asset_raw_segment_relations — 段间有向图，含 relation_type / weight / confidence / distance
STAGE 6 · BUILD_RETRIEVAL_UNITS

生成检索单元

输入Segments + Relations
变换每个 segment 生成多种检索载体，适配不同检索策略
输出

SECTION 08

LLM Service — 统一智能增强层

独立 FastAPI 服务（端口 8900），统一管理所有 LLM 调用。Mining 和 Serving 不直接调用 LLM，通过本服务获取能力。已生产就绪。

核心能力

  • 双模式调用：Mining 异步 submit（批量），Serving 同步 execute（在线）
  • 模板引擎：Jinja2 模板 + JSON Schema 输出校验，模板版本化管理
  • 任务生命周期：queued → running → succeeded / dead_letter，自动重试
  • 幂等控制：idempotency_key 防重复提交
  • 全链路审计：6 张表记录完整的 prompt → attempt → result 链
  • 多 Provider：DeepSeek / OpenAI / 通义千问 / Ollama，可扩展

重试机制：最大 3 次重试，指数退避（backoff_base^attempt_no，最大 60s）。每次尝试记录 token 用量和延迟。同步超时不失败，任务保持 running 状态由 lease recovery 处理。79 个测试用例全覆盖。

SECTION 10

v1.2 目标态 — 知识自主演化

从"被动知识库"演进为"主动知识生态"。五大架构创新，让知识体系能够自我积累、自我校正、自我演进。

核心愿景：v1.1 解决了"如何把文档变成可检索的知识"。v1.2 要解决"如何让知识体系持续演化——自动积累高质量内容、淘汰过时内容、检测矛盾内容、理解段间深层语义"。

创新一：五层本体模型

传统知识库使用扁平本体（实体→属性），无法区分不同认知层次的知识。我们设计了五层递进结构：

L1 概念层
可配置对象
接口、协议、策略、VPN
→
L2 机制层
协议算法
事物如何工作
→
L3 方法层
操作程序
怎么配置、怎么排障
→
L4 条件层
决策规则
什么场景用什么方案
→
L5 场景层
业务场景
实际部署上下文
为什么五层：不同角色需要不同粒度的知识。集成工程师关注"怎么配"（方法+场景），方案设计师关注"什么条件下选什么方案"（条件+机制），架构师关注"概念间的关系"。扁平本体把这四类知识混在一起，查询粒度无法控制。

创新二：双层图架构

传统做法把事实直接连到知识图谱节点，导致图既用于推理又用于溯源，互相干扰。我们创新性地采用双层分离：

推理层 — 本体图

节点间只有语义关系边（DEPENDS_ON、EXPLAINS 等）。这些边是多条事实蒸馏后的聚合结论。

  • 3 条事实只产生 1 条边 → 图保持简洁
  • 支持依赖闭包、影响传播、跨层推理

溯源层 — 证据链

Fact → Evidence → Segment → Document 的完整证据链，回答"这个结论从哪来"。

  • 事实不直接连到本体节点（无 ABOUT_SUBJECT 边）
  • 事实生命周期管理完全独立
  • 推理查询只走推理层，溯源查询只走溯源层

创新三：声明式查询引擎

传统知识库要么只支持关键词搜索，要么需要复杂编程。我们设计了 5 个代数原语，用户只需表达"要什么"，规划器自动优化执行路径：

原语功能示例
seed初始化结果集按 ID / 别名 / 层级 / embedding / 属性匹配
expand图遍历沿本体边扩展，支持 tagged_in / rst_adjacent / evidenced_by
combine集合操作union / intersect / subtract
aggregate聚合排序分组、评分、关键词相关性、可选 Cross-Encoder 重排
project字段投影选择输出字段，组装最终 ContextPack
自动并行：规划器检测步骤间的变量依赖，将无依赖步骤编组为并行执行波，总查询超时 30s。

创新四：知识自治演化 — Auto-Promote 机制

知识体系不是静态的，而是持续自我改进的。通过三层候选词治理 + 自动晋升门控实现"知识的自然选择"：

Stage 3 文档级：每文档 LLM 分类 + 停用词过滤 + 初始评分 → 产出候选词
Stage 3b 循环级：基于累积 source_count 轻量重评分 → 高频词得分上升
Maintenance 日级：embedding 聚类去重 + LLM 批量分类 + 人工终审 → 最终确认
Auto-Promote 门控：当候选词满足 score ≥ 0.85 + 所有门控通过 + 存在 ≥7 天 → 自动晋升到本体。高质量知识自动积累，低质量被淘汰。

创新五：五维置信度评分

每条知识都有量化可信度评估，不是简单的"相关度分数"，而是五个维度的加权综合：

维度权重说明
source_authority 来源权威度0.30电信领域来源质量差距巨大（3GPP 规范 vs 论坛帖子），权重最高
extraction_method 提取方法0.20专门设计的规则（0.85）比通用 LLM（0.70）更可靠
ontology_fit 本体契合度0.20与已有本体结构的匹配程度，防止漂移
cross_source_consistency 跨源一致性0.20多个来源是否一致，冲突内容降分
temporal_validity 时效性0.10过时内容随时间自然衰减
v1.2 技术栈总览：五层本体 + 双层图 + 声明式查询 + 自治演化 + 五维置信度 + 语篇关系感知分段 + 父子层级检索 + 上下文增强 + BGE-M3 中文 embedding + jieba 分词。这不是功能的堆砌，而是一套互相增强的完整体系。
